# 心动实验室 — 产品需求文档

> 这份文档描述产品的设计意图：隐藏状态如何演化、结局如何判定、为什么坚持不给玩家看进度条。
> 想理解「这游戏为什么这样设计」看这里；想跑起来看根目录 [README](../README.md)。
> 文档与代码冲突时以代码为准。

---

## 1. 产品定位

心动实验室是一款基于 AI 人设生成、长期关系对话、隐藏状态演化与复盘分析的关系对话模拟产品。

用户可以创建一个目标人设，与其展开长期聊天互动。系统在对话过程中持续更新角色对用户的关系判断，并以"心动值"作为唯一外显反馈。用户可在满足冷却条件后发起复盘，获得对既往对话的分析与建议。

这个产品不是"帮用户代聊"，不是"提供标准答案"，不是"教人套路"。它的本质是通过模拟真实关系中的模糊反馈、长期演化与不确定性，让用户在体验中逐步理解：关系如何建立，关系如何聊坏，节奏、边界与表达方式如何影响关系走向。

---

## 2. 核心目标

第一阶段的核心目标是做成一个**以真实沉浸模拟为主、以复盘分析为辅的关系对话系统**。

用户在产品中的主目标为：与角色建立长期互动，在不确定反馈中逐步判断关系走向，在合适时机表达心意，由系统根据长期互动自然判定关系结果。

产品的底层价值观是：恋爱不是靠死追、硬推、讨好、套路，吸引来自自然表达、边界感、真实感、自我感与节奏感。产品不教用户"拿下对方"，而是帮助用户理解为什么会把关系聊坏、为什么越努力越让人下头、什么叫舒服自然不过度推进的交流。

产品的目标路径是**"通过 A 最终找到 B"**：

- **A** = 让用户自己去聊、去经历、去犯错、去感受
- **B** = 理解"她喜不喜欢你，其实取决于你自己和你的聊天模式"

不能跳过 A 直接把 B 告诉用户，不能上来就教育用户，必须让用户在互动中自己感受到这个结论。这是一个**体验式认知产品**，不是说教型产品。

---

## 3. 产品调性原则

- 不直接教套路，不提供标准化"拿下"答案
- 不把复盘作为金科玉律，复盘仅提供一种理解与思路
- 不主动宣告"你失败了"，让用户通过长期反馈自行感知
- 不允许从中间节点回档重聊
- 核心人格保持稳定，关系状态和表达方式随互动演化
- 同一人设可分化出多个长期独立角色，不同角色发展互不影响

## 4. 核心概念定义

### 4.1 人设（Persona）

人设是角色创建前的初始模板，用于定义一个目标对象的基础属性，包括基础身份信息、性格标签、表达风格、兴趣偏好、初始关系倾向、初始隐藏判断等。人设本身不是长期互动对象，而是角色生成的来源。

### 4.2 角色（Character）

角色是用户基于某个人设开启对话后形成的独立实例。一旦进入对话，系统会为该实例持续累积聊天记忆、关系状态、对用户的印象变化、表达方式变化与终局可能性演化。同一人设可派生多个角色实例，每个角色实例都是独立存档、独立记忆、独立发展线。虽然起点相同，但因为聊天过程不同，最后会长成不同角色。

### 4.3 心动值（Heartbeat Score）

心动值是用户可见的唯一外显关系反馈值。它不等同于真实好感度，也不代表最终结果，而是系统对当前关系互动状态的综合映射。系统底层由多个隐藏维度共同驱动心动值变化，但这些维度不对用户展示。前台只给用户看一个单值，不展示复杂维度解释，不做太明显的系统提示。

### 4.4 复盘（Review）

复盘分为短复盘和长复盘。短复盘面向最近若干轮对话，长复盘面向整段关系轨迹。复盘不会改变角色状态，也不会成为"正确答案"，其作用是帮助用户理解已有互动中的问题、节奏和可能风险。复盘有冷却机制，不是每轮都能点。复盘口吻采用**成熟关系观察者风格**，不是锐利教练，不是高高在上的评判机器。

### 4.5 终局（Ending）

终局指某条角色关系线进入不可继续推进的明确收束状态。终局包含四种情况：明确表白并成功、明确表白且系统判定关系不可继续、用户主动归档或放弃角色、因 **§4.6** 所述会话与时间边界触发而导致的强制收束终局。长期低状态但未明确终止不视为终局，角色仍会持续存在（与第四种「明确规则触发」的收束相区分）。

### 4.6 会话与时间边界

单角色关系线需具备**可解释的**聊天、会话或进度边界，避免无期限、无边界的消耗。达到该边界后，该关系线**不可继续对话**，并进入**终局结算**（与长复盘、终局评价等能力的衔接方式随实现演进）。

时间基准可采用现实时间、游戏内时间或混合方案，不在本文写死。

本机制针对用户在**活跃使用**或产品内**约定进度**下的收束；**不等于**「用户长期不上线即自动删档或静默结束」。

详细开放问题与实现挂点见 [TIME_SYSTEM_PLAN.md](TIME_SYSTEM_PLAN.md)。

## 5. 用户故事

**用户故事一：** 用户对一个真实女生有聊天困惑，希望通过上传聊天记录生成一个尽可能接近她的角色，与之继续互动，观察不同聊天方式会导致怎样的关系变化。

**用户故事二：** 用户没有具体对象，但想进入一个高沉浸的关系模拟场景，通过聊天体验不同性格对象的反馈，并理解自己的聊天模式。

**用户故事三：** 用户已经和某角色聊了很久，心动值一直不上升，希望通过复盘分析哪里出了问题，但系统不会直接告诉他"已经失败"。

## 6. 产品范围

### 6.1 第一阶段必须包含

- 人设创建（对话式创建）
- 角色实例化
- 长期聊天
- 心动值显示
- 隐藏状态演化
- 短复盘
- 长复盘
- 表白自动识别
- 终局判定
- 可解释的会话/时间边界与到时收束、终局结算（规则细节见 `TIME_SYSTEM_PLAN.md`，随开发落地）
- 人设库 / 角色列表 / 会话存档

### 6.2 第一阶段不做

- 多人协作围观
- 分享海报
- 排行榜
- 角色语音 / 视频
- 用户回档重试
- 基于时间的状态衰减（指心动值、关系状态等随墙钟、无人互动而「自动下滑」；**不**包含 §4.6 所指之会话边界、游戏内时间推进、时长或回合上限等产品规则）
- 复盘结果直接改写当前会话
- 明确展示隐藏阶段、隐藏判断、底层多维数值

## 7. 系统对象与关系

### 7.1 系统对象


| 对象                 | 说明                          |
| ------------------ | --------------------------- |
| 用户（User）           | 发起人设创建、角色创建、对话、复盘、归档        |
| 人设（Persona）        | 作为角色生成模板存在，可重复被用于创建多个角色实例   |
| 角色（Character）      | 绑定某个用户与某个人设来源，具有独立会话和独立关系状态 |
| 对话会话（Conversation） | 某个角色下的完整聊天记录                |
| 复盘记录（Review）       | 短复盘 / 长复盘的生成结果              |
| 终局结果（Ending）       | 当角色进入终局时的最终结果说明与分析          |


### 7.2 对象关系

- 1 个用户可拥有多个人设
- 1 个人设可生成多个角色
- 1 个角色对应 1 条长期独立关系线
- 1 个角色包含 1 组持续更新的隐藏状态
- 1 个角色可多次触发短复盘，并在终局时生成长复盘与终局评价

## 8. 核心流程

### 8.1 流程一：创建人设

用户通过与人设创建智能体对话来完成人设创建。用户可以用自然语言描述目标对象的任意信息，也可以上传聊天记录等多模态素材。智能体在对话过程中会根据用户提供的信息逐步构建人设，并在信息不足时主动提问引导用户补充，帮助完善人设设定。创建过程中，可见层信息实时展示供用户确认，隐藏层信息用户不可查看，智能体也不得主动透露。

### 8.2 流程二：生成人设档案

创建完成后，系统为该人设生成两层内容：

**用户可见层：**

- 昵称 / 称呼
- 基本身份
- 性格简介
- 兴趣点
- 聊天风格描述
- 可见背景信息

**系统隐藏层：**

- 初始倾向
- 初始印象基础值
- 对用户的关键判断
- 对推进节奏的容忍度
- 对某类表达的敏感点
- 底层关系演化参数

隐藏层信息分为两类：用户主动设定的和系统自动生成的。用户主动设定是指用户在创建对话中明确提出了涉及隐藏层的需求（如"创建一个对我有好感的女生"、"她一开始应该对我比较冷淡"），此时智能体应将需求写入隐藏层，并用自然语言做模糊确认（如"好的，她一开始会对你有不错的印象"），但不暴露具体参数和数值。系统自动生成是指用户未提及、由系统根据整体人设自动推算填充的隐藏层字段，智能体不得以任何形式向用户透露这些信息。如用户在创建过程中主动询问系统自动生成的隐藏层内容（如"她喜不喜欢我"），智能体应以自然方式拒绝回答（如"这个要等你们真正聊起来才能感受到"），不得直接说"这是隐藏信息"。

### 8.3 流程三：从人设派生角色

用户点击"开始聊天"后，系统基于当前人设生成一个角色实例。该角色继承人设的基础设定，拥有独立记忆、独立关系状态，成为后续所有对话与复盘的唯一主体。同一个人设可被多次启动，形成多个不同角色实例。

### 8.4 流程四：长期对话

用户进入角色对话页后，与角色进行持续聊天。每轮聊天后系统更新两类信息：

**用户可见：** 心动值

**用户不可见：** 关系阶段、对用户的即时判断、长期印象变化、底层多维状态、当前是否处于难修复区间、是否存在表白触发信号

角色会在长期互动中演化：核心人格不变，面对用户的表达方式会变化，记忆不断累积，回应风格会随关系变化而变化。

### 8.5 流程五：短复盘

在满足冷却条件后，用户可触发短复盘。短复盘针对最近若干轮对话输出：

- 用户哪些表达可能存在问题
- 这些问题可能造成了什么关系影响
- 有哪些更自然或更稳妥的改进思路
- 当前阶段用户最值得注意的问题是什么

短复盘不修改角色状态，不触发回档，不自动改变角色判断。

### 8.6 流程六：关系长期演化

角色对用户的态度不会被系统直接说死，但会通过心动值长期走势、角色回应方式、对话流动性、短复盘提示，让用户逐渐意识到关系在升温、停滞、变冷、进入高修复成本区，或者自己的聊天方式正在产生长期影响。

如果角色长期进入无法修复的低状态：系统不主动宣告结束，角色仍存在，用户仍可继续聊天，但关系不再自然推进。**失败是被感受到的，不是被系统直白告知的。**

### 8.7 流程七：表白自动识别

系统在对话中持续识别用户输入属于哪个类别：

**轻表达** — 如"我有点喜欢你""感觉和你聊天挺开心""我好像越来越在意你了"。轻表达不直接进入终局判定。

**明确关系确认** — 如"我喜欢你，想和你在一起""你愿不愿意做我女朋友""我想正式追你/跟你确认关系"。当系统识别为明确关系确认时，进入终局判定流程。

### 8.8 流程八：终局判定

明确关系确认后，系统结合当前心动值、长期关系走势、历史互动轨迹、底层隐藏维度、初始倾向、当前角色对用户的判断、是否已进入可表达窗口等因素做判定。

**结果 A：表白成功**

该角色关系线终局结束，系统生成终局评价、长复盘与关系风险分析。即使成功，也可能被评估为"关系存在风险"，例如关系推进方式不健康、用户更多依赖技巧而非真实连接、角色虽然接受但长期信任基础不足。终局不是"赢了就赢了"，结束后系统仍会从关系质量角度给出评价。

**结果 B：表白失败但关系可继续**

角色关系线不结束，继续存在。系统不给出"终局失败"，只体现为角色回应与后续关系状态变化。

**结果 C：表白失败且关系不可继续**

该角色关系线终局结束，系统输出失败后的终局评价与复盘。

### 8.9 流程九：长复盘与终局评价

长复盘面向整段关系轨迹，主要输出：

- 用户整段关系中的主要问题
- 用户在哪些阶段做对了 / 做错了
- 用户的长期聊天模式特点
- 角色为什么会形成如今的终局判断
- 如果成功，关系未来可能的风险点
- 如果失败，关键转折点在哪里

复盘口吻采用**懂关系的成熟旁观者视角**，不是标准答案，不是打分老师，不是套路教练。

## 9. 隐藏状态系统

### 9.1 设计原则

系统底层采用多维隐藏状态驱动关系演化，第一阶段不向用户展示这些维度名称和具体数值。

### 9.2 底层维度

内部至少包含以下维度：


| 维度     | 说明                 |
| ------ | ------------------ |
| 舒适感    | 角色在与用户互动中的放松程度     |
| 兴趣度    | 角色对用户话题与表达的兴趣水平    |
| 信任感    | 角色对用户意图与真实性的信任程度   |
| 警惕度    | 角色对用户行为的防备程度（越高越差） |
| 初始匹配基线 | 人设层面预设的与用户的基础契合度   |


这些维度共同影响：心动值、角色表达方式、对用户的印象变化、是否进入难修复区间、是否具备关系确认可能。

### 9.3 角色演化原则

- 核心人格稳定
- 关系状态可演化
- 表达方式可演化
- 对用户的判断可演化
- 初始倾向存在，但终局仍有演化空间

## 10. 信息架构与页面

### 10.1 首页主视角

首页主视角为**角色对话列表**。用户打开产品后优先看到自己正在进行或已存档的角色关系线，而不是抽象人设模板。

### 10.2 首页内容

**角色列表区** — 每个角色卡片包含：

- 角色名称
- 来源人设名称
- 当前心动值
- 最近一条消息摘要
- 当前状态标签（进行中 / 已结束 / 已归档）
- 更新时间

**人设入口区** — 创建新角色、查看 / 管理人设库

### 10.3 人设库页

展示用户已创建的人设模板，支持查看人设简介、基于人设启动新的角色实例、删除或归档人设。

### 10.4 角色详情页 / 对话页

包含可见角色简介、心动值展示、聊天区、输入区、复盘入口、角色状态基础标识、历史会话持续保留。

### 10.5 复盘页

展示短复盘结果、历史复盘记录、长复盘或终局复盘。

### 10.6 第一阶段页面清单


| 页面           | 说明             |
| ------------ | -------------- |
| 首页           | 角色对话列表页        |
| 人设创建页        | 创对话式创建界面       |
| 人设生成结果页      | 展示生成的人设档案（可见层） |
| 人设库页         | 已有人设管理         |
| 角色对话页        | 核心聊天界面         |
| 短复盘弹窗 / 页面   | 短复盘结果展示        |
| 终局结果页        | 终局评价与分析        |
| 长复盘页         | 整段关系复盘         |
| 角色归档 / 结束确认页 | 简单确认操作         |

- 人设 ID
- 创建方式
- 可见信息（昵称、身份、性格、兴趣、风格、背景）
- 隐藏信息（初始倾向、印象基础值、关键判断、节奏容忍度、敏感点、演化参数）
- 生成原始依据（聊天记录 / 描述文本）

### 12.2 角色层

- 角色 ID
- 来源人设 ID
- 当前状态（进行中 / 已结束 / 已归档）
- 当前心动值
- 隐藏状态快照
- 是否已结束
- 是否已归档

### 12.3 对话层

- 消息历史
- 轮次
- 表白识别标记
- 阶段性状态变更记录（内部）

### 12.4 复盘层

- 短复盘记录
- 长复盘记录
- 触发时间
- 所分析的会话范围

## 13. 关键规则边界

### 13.1 用户不可做

- 不能从中间节点回档重聊
- 不能看到隐藏判断
- 不能看到底层多维数值
- 不能通过复盘直接改写当前角色状态

### 13.2 系统不可做

- 不主动弹窗宣告"失败"
- 不把复盘说成唯一正确答案
- 不因长时间不聊天自动结束角色
  - 说明：该条仅约束**冷置**（用户长期无互动）不因「久未打开」而自动结束角色实例；**不禁止**依据 §4.6，因会话内可解释规则（如进度、时长、游戏内日等）触发的关系线收束与终局结算。
- 不让角色核心人格被彻底聊变

## 14. 第一阶段验收标准

### 14.1 基础功能验收

- 能成功通过对话式创建完成人设
- 能基于同一人设创建多个独立角色
- 多个角色记忆、状态、结果相互独立
- 角色对话可长期保存
- 心动值在对话中会动态变化
- 复盘有冷却限制且能生成有效分析
- 系统可自动识别"轻表达"和"明确关系确认"
- 明确关系确认后，系统能生成不同结果
- 表白成功后可生成终局评价
- 长期低状态不会被系统直接宣告失败
- 当 §4.6 所定规则生效时，用户能感知边界存在，且在触发后进入终局或结算相关流程（具体交互与文案随迭代，不在本 PRD 写死）

### 14.2 体验验收

- 用户能明显感受到同一人设在不同角色线下会发展出不同结果
- 用户能通过角色回应和心动值变化感知关系升温 / 停滞 / 变冷
- 复盘内容不是模板化空话，而是能对具体对话提出洞察
- 产品整体更像真实关系模拟，而不是简单刷分聊天工具
